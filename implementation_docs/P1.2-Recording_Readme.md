# Aadhavi Essentials: CCaaS Audio Recording & QA Architecture

## 📌 Phase 2 Executive Summary
This document outlines the Phase 2 implementation of the Aadhavi Essentials CCaaS middleware. Building upon the core Skills-Based Routing (SBR) engine, this phase introduces enterprise-grade Call Recording and Quality Assurance (QA) capabilities. 

By leveraging Twilio's asynchronous webhooks and Salesforce's REST API, the system now captures live, two-way conversational audio in stereo format and automatically attaches the resulting MP3 URL directly to the routing Case. This ensures all customer-agent interactions are natively logged within the CRM without requiring third-party storage interfaces.

---

## 🏗️ Architectural Enhancements

* **Asynchronous Audio Processing:** Twilio processes the audio file post-call and fires a secondary, delayed webhook to the NestJS server.
* **Dual-Channel Recording:** Utilizes `record-from-answer-dual` to separate the customer audio (Left Channel) and executive audio (Right Channel), optimizing the data for future AI transcription.
* **Idempotent Data Updates:** The NestJS middleware performs a `PATCH` request to the Salesforce Case, ensuring the audio URL is appended without overwriting existing IVR data (like Customer IDs).

---

## ⚙️ Implementation Steps

### 1. Salesforce Database Expansion
* Created a custom URL field on the `Case` object with the API Name `Call_Recording_URL__c`.
* **UI Configuration:** Exposed the field to the support team by modifying the Lightning Record Page (Dynamic Forms) to pull the database value into the active Service Console layout.

### 2. NestJS Middleware Refactoring
* **Synchronous Update:** Modified the existing OAuth routing sequence to `await` and return the newly generated `CaseId`.
* **New Patch Method:** Created `attachRecordingToCase(caseId, recordingUrl)` in the Salesforce Service to handle the secondary data append.
* **New Webhook Endpoint:** Configured the Twilio Controller to expose a `/twilio/recording-done` POST endpoint specifically to catch Twilio's asynchronous file delivery.

### 3. Telecom Edge (Twilio TwiML)
* Upgraded the `<Dial>` verb to include recording parameters.
* **Payload Structure:** `<Dial callerId="+1xxx" record="record-from-answer-dual" recordingStatusCallback="[NGROK_URL]/twilio/recording-done?caseId=[SF_CASE_ID]">[AGENT_PHONE]</Dial>`

## 🛑 Real-World Challenges & Architectural Solutions
*Note for Interviews: These represent actual implementation hurdles encountered during the recording phase.*

### 1. The `INVALID_FIELD` Database Error
* **Issue:** The NestJS API returned `No such column 'Call_Recording_URL__c' on sobject of type Case`.
* **Root Cause:** Salesforce operates on a "Zero Trust" model. Even if a field exists, the API execution user ("Run As" user) must have explicit Field-Level Security (FLS) profile permissions to read/write to it.
* **Solution:** Navigated to Object Manager -> Case -> Fields -> Set Field-Level Security, and explicitly checked the "Visible" permission for the System Administrator profile, granting the Connected App write access.

### 2. The Data vs. UI Disconnect
* **Issue:** The backend logged a successful `PATCH` request (`Successfully attached audio recording...`), but the URL was completely invisible on the Salesforce Case screen.
* **Root Cause:** The CRM backend and frontend are decoupled. Saving data to a table does not automatically paint it onto the user interface.
* **Solution:** Bypassed classic Page Layouts and edited the Lightning Record Page directly, dragging the `Call Recording URL` component onto the active screen to visualize the stored database value.

### 3. SOQL Syntax Limitations
* **Issue:** Running `SELECT ALL FROM Case WHERE Id = '...'` to verify the database resulted in a syntax error.
* **Root Cause:** Unlike standard SQL (PostgreSQL/MySQL), Salesforce Object Query Language (SOQL) aggressively restricts `SELECT *` commands to prevent multi-tenant server degradation from unoptimized queries.
* **Solution:** Rewrote the query to explicitly request necessary columns: `SELECT Id, Subject, Call_Recording_URL__c FROM Case WHERE Id = '...'`.
