# CCaaS Architecture: Salesforce Service Cloud Voice & Amazon Connect

## Executive Summary
This document outlines the end-to-end implementation of a Cloud Contact Center as a Service (CCaaS) integration. The architecture bridges Amazon Connect (Telephony/Media) with Salesforce Service Cloud Voice (Signaling/CRM) using the Partner Telephony model. The solution delivers a unified agent experience through a native WebRTC softphone embedded within the Salesforce Omni-Channel utility.

## Technology Stack
* **CRM & Agent Workspace:** Salesforce Service Cloud Voice (SCV)
* **Telephony Backbone & IVR:** Amazon Connect (`aadhavi-ccaas-hub`)
* **Middleware & Event Triggers:** AWS Lambda (Serverless)
* **API Framework:** Salesforce Telephony Integration API
* **Client-Side Media:** WebRTC, Amazon Contact Control Panel (CCP)

---

## 1. The Telephony & Media Layer
**Objective:** Establish a high-fidelity, browser-based voice connection without requiring external telephony applications.

* **CTI Integration:** Deployed the native SCV framework rather than relying on standard managed packages. This allows Salesforce to treat voice as a native routing object rather than a siloed application.
* **WebRTC Softphone:** Configured the Omni-Channel widget to act as the primary agent interface. 
* **Implementation Example:** Successfully separated the "Signaling Layer" from the "Media Layer." When browser pop-up blockers initially prevented the WebRTC media stream from establishing, the signaling layer still successfully generated a `Voice Call` (VC-000001) record. Resolved the media failure by authorizing persistent cross-site microphone and pop-up access for the CCP hidden window.

## 2. Orchestration & Routing Engine
**Objective:** Design a seamless customer journey from inbound dial to agent connection.

* **IVR & Contact Flows:** Architected the front-end Amazon Connect Contact Flow to handle inbound digit collection (e.g., "Press 1 for Sales") and set the working queue variable dynamically.
* **Omni-Channel Configuration:** Configured Salesforce Presence Statuses (Available - Voice) and Routing Configurations to dictate capacity and agent availability.
* **Queue Mapping:** Bridged the two cloud environments by statically mapping Amazon Connect Queues to Salesforce Queues.
* **Implementation Example:** Configured the AWS IVR `Set Working Queue` block to target the `BasicQueue`. Built the integration map in Salesforce to ensure any payload originating from the AWS `BasicQueue` was routed natively into the `SCV Basic Queue` for Omni-Channel distribution.

## 3. Integration & Middleware Layer
**Objective:** Facilitate real-time, bidirectional communication between AWS and Salesforce.

* **Serverless Event Middleware:** Utilized AWS Lambda functions deployed via the SCV wizard. When a call hits the "Transfer to Queue" block in AWS, a Lambda function is triggered.
* **Telephony API:** The Lambda function authenticates into Salesforce via a Connected App and posts the call payload via the Telephony API. This is the mechanism that translates an AWS queue event into a ringing notification on the agent's screen.
* **Single Sign-On (SSO) & Identity:** Navigated complex authentication flows to ensure agents maintain concurrent, secure sessions in both platforms.
* **Implementation Example:** Debugged a critical SSO token handoff failure during provisioning. Browsers utilizing strict privacy settings (blocking third-party cookies) destroyed the OAuth handshake between the identity provider and the target domain. Resolved by establishing explicit domain exceptions (`[*.]force.com` and `[*.]salesforce.com`) to allow the secure tokens to pass.

## Architectural Challenges & Resolutions (Troubleshooting Log)

| Component | Issue | Root Cause | Resolution |
| :--- | :--- | :--- | :--- |
| **Routing** | Unable to map standard queues to AWS. | "Enhanced Omni-Channel Routing" enforced flow-based routing, locking the static mapping UI. | Reverted the org's core routing engine to Standard Omni-Channel to allow direct 1:1 queue bridging. |
| **Identity** | Total environment lockout; unable to trigger password reset. | System provisioned without a `PasswordHint` (Null value), causing the reset JSP page to crash. | Bypassed the reset UI by using the direct Master OAuth Token via the environment's launch portal, followed by an immediate internal password rotation. |
| **Media Layer** | Agent accepted call, but caller remained on hold. | The Omni-Channel API signaled the answer, but the browser blocked the WebRTC audio window. | Authorized cross-site pop-ups and validated hardware microphone permissions against the AWS CCP domain. |
